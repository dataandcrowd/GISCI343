Why we package code

Imagine you wrote a walkability function for Project A.

# project1/analysis.py
def calculate_walkability(streets, amenities, buffer_m=800):
    # 50 lines of careful code
    ...

Six months later, Project B needs the same function. What do you do?

Most people copy-paste. Then they fix a bug in Project A and forget Project B. Now there are two versions of “the same” function and neither matches.

Packaging in one slide

# Once: build and publish
pip install my-urban-toolkit

# Forever: import in any project
from my_urban_toolkit import calculate_walkability

result = calculate_walkability(streets, amenities)

One source, and one install command. Every project (and every collaborator) gets the same function.

I used to teach OOP in a hard way…

  • Encapsulation is the process of making data private by wrapping data and its methods in a ‘capsule’ or unit
  • It can not be accessed or modified outside of that unit

Real-World Example of Encapsulation:

  • Imagine you’re building software for a coffee shop. You might have a class that represents a coffee machine.
  • The coffee machine has many internal components, like water levels, temperature settings, and a coffee bean grinder.
  • water_level, temp, set_grind_for_espresso()

What is a Python package, really?

  • A module is a single .py file.
  • A package is a folder of modules with an __init__.py.
  • A distribution is the .whl or .tar.gz file you install.
  • A PyPI release is a distribution uploaded to https://pypi.org so pip install can find it.

Today we go: file → folder → distribution. Publishing to PyPI is Week 12.

Anatomy of a package: the src/ layout

geohello-yourname/                # The repository
├── src/
│   └── geohello_yourname/        # The Python package (underscores)
│       ├── __init__.py           # Makes the folder importable
│       └── py.typed              # Optional type-checker marker
├── tests/                        # pytest finds these next week
├── pyproject.toml                # Package configuration
├── README.md                     # What people see on PyPI
└── .gitignore

Two names to keep apart:

  • PyPI name (in pyproject.toml): hyphenated, e.g. geohello-yourname
  • Python package name (folder under src/): underscored, e.g. geohello_yourname

pip install geohello-yourname then from geohello_yourname import hello.

pyproject.toml, the configuration hub

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "geohello-yourname"
version = "0.1.0"
description = "A simple geospatial greeting"
readme = "README.md"
requires-python = ">=3.10"
dependencies = []

Two sections. [build-system] says how to build. [project] says what the package is. We add dependencies and [project.optional-dependencies] later.

Live demo: four steps to a working package

We will build geohello together. Same commands you will run in pairs in 10 minutes.

# Step 1: scaffold
uv init --lib geohello
cd geohello

# Step 2: edit src/geohello/__init__.py (one function)

# Step 3: build
uv build

# Step 4: try it
uv run python -c "from geohello import hello; print(hello())"

Step 1: scaffold with uv

uv init --lib geohello

This creates the whole folder for you.

geohello/
├── src/geohello/__init__.py
├── tests/
├── pyproject.toml
├── README.md
└── .gitignore

uv already filled in a working pyproject.toml and made the src/ layout correctly. You did not have to remember any of it.

Step 2: write one function

Open src/geohello/__init__.py and replace the contents:

"""A simple geospatial greeting package."""

__version__ = "0.1.0"


def hello(place="Auckland"):
    """Return a greeting for a place."""
    return f"Hello from {place}!"

That is the entire package. One function, one docstring, one version.

Step 3: build the distributions

uv build

This creates a dist/ folder.

dist/
├── geohello-0.1.0-py3-none-any.whl    # the wheel
└── geohello-0.1.0.tar.gz              # the source archive

The wheel is what pip install actually downloads. The .tar.gz is a backup that contains your source code.

Step 4: try it

# Run the function in one line
uv run python -c "from geohello import hello; print(hello())"
# Hello from Auckland!

# Or open Python interactively
uv run python
>>> from geohello import hello
>>> hello("Tokyo")
'Hello from Tokyo!'

That is what “building a package” feels like. Four commands.

Common errors and fixes

  • ImportError: No module named geohello_yourname. Check the folder under src/ exists and uses underscores.
  • name 'geohello' is already taken (later, on TestPyPI). Add your initials to the name field.
  • Build fails on the first line of pyproject.toml. Usually a TOML syntax typo. Read the first error message carefully.
  • The function works in uv run python but not in plain python. You are not in the project’s virtual environment. Use uv run or source .venv/bin/activate.

A real package: akl-ped-counts

github.com/dataandcrowd/akl-ped-counts

A small package on PyPI that ships hourly Heart of the City pedestrian counts (21 sensors, 2019 to 2025).

pip install akl-ped-counts
from akl_ped_counts import load_hourly, load_locations, list_sensors

print(len(list_sensors()))   # 21
counts = load_hourly()
print(counts.shape)          # (61367, 24)

Three functions, one short README, one PyPI release. This is the size and shape your A3 should aim for.

What’s inside the repo

akl-ped-counts/
├── pyproject.toml
├── README.md
├── LICENSE
├── CONTRIBUTING.md
├── examples/
│   └── march_trajectories.png
└── src/akl_ped_counts/
    ├── __init__.py            # Re-exports the public API
    ├── loader.py              # The actual functions
    ├── polars_loader.py       # Optional Polars backend
    └── data/                  # Bundled CSV files

The patterns we just covered are all there. src/ layout, thin __init__.py, hatchling backend, optional dependencies for [plot] and [geo].

Optional dependencies, an idea worth copying

[project]
dependencies = ["pandas>=1.5.0"]   # core only

[project.optional-dependencies]
polars = ["polars>=0.20.0"]
plot = ["matplotlib>=3.5.0", "seaborn>=0.12.0"]
geo = ["folium>=0.14.0"]
all = ["polars>=0.20.0", "matplotlib>=3.5.0",
       "seaborn>=0.12.0", "folium>=0.14.0"]

A user who only wants the data installs pip install akl-ped-counts. A user who wants plotting installs pip install "akl-ped-counts[plot]". The core install stays small.

Part 7: Where does the data live?

Two patterns

Your A3 package needs data. Real Auckland data, not synthetic toys. Two patterns to choose from.

  • Bundle it. Ship a small file inside the package itself. Good for lookup tables, schemas, sensor lists, anything under about 10 MB.
  • Fetch it. Download from an API on first use, then cache locally. Good for large or frequently updated data such as Stats NZ census and LINZ boundaries.

akl-ped-counts uses the first pattern (8 MB of CSVs ship with the package). Most A3 briefs will use the second.

Pattern 1: Bundling GIS data

Where to put the file

The convention is a data/ folder inside the package, not at the top level.

your-package/
├── pyproject.toml
├── README.md
└── src/
    └── your_package/
        ├── __init__.py
        ├── loader.py
        └── data/                # <-- here
            ├── sensors.csv
            └── zones.geojson

This is what akl-ped-counts does with hourly_counts.csv and locations.csv. Same idea, same place.

Finding the file at runtime

Do not hard-code paths like "/Users/me/Desktop/sensors.csv". The file moves when the package is installed.

Use importlib.resources instead.

from importlib import resources
import geopandas as gpd
import pandas as pd


def _data_path(filename: str) -> str:
    """Resolve a path to a bundled data file."""
    ref = resources.files("your_package") / "data" / filename
    return str(ref)


def load_sensors():
    return pd.read_csv(_data_path("sensors.csv"))


def load_zones():
    return gpd.read_file(_data_path("zones.geojson"))

This works whether the package is installed from PyPI, from a wheel, or in editable mode.

Telling the build tool to include the file

Bundled data does not ship by default. You must tell hatchling to include it.

[tool.hatch.build.targets.sdist]
include = [
    "src/your_package/**",
    "README.md",
    "LICENSE",
]

[tool.hatch.build.targets.wheel]
packages = ["src/your_package"]

The ** glob picks up everything under src/your_package/, including the data/ folder. Without this, your CSVs are missing from the wheel and pip install looks fine but breaks at runtime.

Always sanity-check by unzipping the .whl after uv build. The data/ folder should be inside.

When to bundle vs not

  • Yes, bundle. Sensor metadata (a few KB). Schema files. Static reference tables. NZDep2018 score table (a few hundred KB).
  • Maybe bundle. A small CSV under 10 MB that does not change between releases.
  • No, do not bundle. Anything over 10 MB. SA2 boundary GeoPackages (often 100+ MB). LINZ canopy layers. Census shapefiles.

For the heavy stuff, fetch from an API.

Pattern 2: Fetching from the Stats NZ API

Why fetch instead of bundle

  • The data is too large to ship in a wheel.
  • The data updates every few months and you do not want to release a new version each time.
  • Stats NZ already hosts it. Stop re-hosting public data.

The cost is one API call on first use. The benefit is a tiny package and always-fresh data.

Stats NZ Datafinder, in plain language

Stats NZ runs a Koordinates-based portal at https://datafinder.stats.govt.nz/.

Two ways to download programmatically.

  • WFS (Web Feature Service). Standard OGC protocol. Returns GeoJSON. Best for vector layers and small bounding boxes.
  • Koordinates Export API. Asynchronous job, you request an export and poll until it finishes. Best for large or full-country pulls.

We will use WFS in the example because it is one HTTP call.

Step 1, get an API key

  1. Sign up at https://datafinder.stats.govt.nz/.
  2. Go to your profile, then “API keys”.
  3. Create a key and copy it. It looks like 4f1a....
  4. Save it somewhere safe.

You can do this during today’s hands-on. The free tier is plenty for any A3 package.

Step 2, never hard-code the key

import os


def _get_api_key():
    key = os.getenv("STATS_NZ_API_KEY")
    if key is None:
        raise EnvironmentError(
            "Set STATS_NZ_API_KEY in your environment "
            "or .env file before calling this function."
        )
    return key

Then run with the key in the environment.

export STATS_NZ_API_KEY=4f1a...
uv run python -c "from your_package import load_sa2; load_sa2()"

Or add the key to a .env file (and add .env to .gitignore).

Step 3, the WFS download

import requests
import geopandas as gpd
from io import BytesIO

WFS_URL = (
    "https://datafinder.stats.govt.nz/services;"
    "key={key}/wfs"
)


def load_sa2_auckland(api_key=None):
    """Download Auckland SA2 boundaries from Stats NZ."""
    api_key = api_key or _get_api_key()
    params = {
        "service": "WFS",
        "version": "2.0.0",
        "request": "GetFeature",
        "typeNames": "layer-111228",       # SA2 2023 generalised
        "outputFormat": "json",
        "srsName": "EPSG:2193",
        "bbox": "1740000,5900000,1790000,5950000,EPSG:2193",
    }
    r = requests.get(WFS_URL.format(key=api_key),
                     params=params, timeout=30)
    r.raise_for_status()
    return gpd.read_file(BytesIO(r.content))

Step 4, cache so you only download once

from pathlib import Path
import geopandas as gpd

CACHE_DIR = Path.home() / ".cache" / "your_package"


def load_sa2_cached(api_key=None):
    CACHE_DIR.mkdir(parents=True, exist_ok=True)
    cache_file = CACHE_DIR / "sa2_auckland.gpkg"

    if cache_file.exists():
        return gpd.read_file(cache_file)

    gdf = load_sa2_auckland(api_key)
    gdf.to_file(cache_file, driver="GPKG")
    return gdf

First call hits the API. Every later call reads the local file. Your demo notebook stays fast.

Hands-on (10 min): try it once

With your partner, do the following.

  1. Sign up at https://datafinder.stats.govt.nz/ and copy your API key.
  2. In your geohello-yourname package, add a function load_sa2_auckland(api_key) based on the slide above.
  3. Set the key in your shell, export STATS_NZ_API_KEY=....
  4. Run uv run python -c "from geohello_yourname import load_sa2_auckland; print(load_sa2_auckland().shape)".
  5. Confirm you get a GeoDataFrame with a sensible row count (around 60 SA2 zones for the Auckland CBD bounding box).

A note for the briefs

Each A3 brief tells you which pattern fits.

  • equitransport benefits from bundling NZDep2018 (a few hundred KB).
  • treecrown-nz and escooter-akl will fetch large LINZ or Stats NZ layers.
  • pedestrian-exposure will likely fetch OSMnx routes and bundle sample inputs for tests.
  • pavescore ships images via the Mapillary API, so it is mostly fetch plus a small fixture image bundled for tests.

Pick the lightest option that works. A package that ships 200 MB of shapefiles is a package nobody installs.

Brief 1: pedestrian-exposure

Cumulative pedestrian exposure to noise, air pollution, and green space along a walking route.

  • load_route(origin, destination, place) from OSMnx, returns edges in EPSG:2193
  • add_exposure(route_gdf, layer, kind) where kind is "noise", "air_quality", or "green"
  • summarise_exposure(route_gdf, weights=None) returns per-edge values for all three plus a length-weighted summary
  • Demo: a CBD walking route + OSM-based noise proxy + LAWA air quality + Sentinel-2 NDVI
  • Difficulty: intermediate to ambitious (three small pipelines, kept consistent)

Brief 2: equitransport

Transport accessibility by deprivation quintile.

  • load_nzdep(sa2_gdf) joins NZDep2018 scores by SA2
  • compute_access(gdf, metric_col) calculates the metric per polygon
  • equity_summary(gdf) returns quintile means + Gini coefficient
  • Demo: Auckland SA2s + AT GTFS PT-stop counts, choropleth by deprivation
  • Difficulty: intermediate (meshblock-to-SA2 aggregation)

Brief 3: treecrown-nz

Urban tree canopy coverage and route shade.

  • load_canopy(bbox) fetches LINZ canopy polygons
  • canopy_coverage(area_gdf) percentage cover per polygon
  • route_shade(route_gdf, buffer_m) canopy fraction along a route
  • Demo: three Auckland suburbs + shade along one walking route
  • Difficulty: intermediate to ambitious (LINZ API quirks)

Brief 4: pavescore

Pavement condition score from a photo.

  • score_image(path_or_url) returns a PavementResult dataclass
  • score_batch(image_list) returns a summary DataFrame
  • to_geodataframe(results, coords) returns a mappable GeoDataFrame
  • Demo: Mapillary images along Dominion Road or Queen Street
  • Difficulty: ambitious, the most open-ended brief

A simple texture-based scorer that works beats an ambitious classifier that breaks.

Brief 5: escooter-akl

Clean OD flows and geofence violations from raw e-scooter trips.

  • load_trips(path) reads CSV, returns trip GeoDataFrame in EPSG:2193
  • od_flows(trips_gdf, zones_gdf) returns origin-destination matrix by zone
  • geofence_violations(trips_gdf, no_park_gdf) flags trips ending in no-park zones
  • Demo: sample of Auckland CBD trips + SA2 zones + CBD pedestrianised polygons
  • Difficulty: intermediate (messy data, loud failure beats silent failure)
