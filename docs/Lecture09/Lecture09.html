Why we package code

Imagine you wrote a walkability function for Project A.

# project1/analysis.py
def calculate_walkability(streets, amenities, buffer_m=800):
    # 50 lines of careful code
    ...

Six months later, Project B needs the same function. What do you do?

Most people copy-paste. Then they fix a bug in Project A and forget Project B. Now there are two versions of “the same” function and neither matches.

Packaging in one slide

# Once: build and publish
pip install my-urban-toolkit

# Forever: import in any project
from my_urban_toolkit import calculate_walkability

result = calculate_walkability(streets, amenities)

One source, and one install command. Every project (and every collaborator) gets the same function.

I used to teach OOP in a hard way…

  • Encapsulation is the process of making data private by wrapping data and its methods in a ‘capsule’ or unit
  • It can not be accessed or modified outside of that unit

Real-World Example of Encapsulation:

  • Imagine you’re building software for a coffee shop. You might have a class that represents a coffee machine.
  • The coffee machine has many internal components, like water levels, temperature settings, and a coffee bean grinder.
  • water_level, temp, set_grind_for_espresso()

What is a Python package, really?

  • A module is a single .py file.
  • A package is a folder of modules with an __init__.py.
  • A distribution is the .whl or .tar.gz file you install.
  • A PyPI release is a distribution uploaded to https://pypi.org so pip install can find it.

Today we go: file → folder → distribution. Publishing to PyPI is Week 12.

Anatomy of a package: the src/ layout

geohello-yourname/                # The repository
├── src/
│   └── geohello_yourname/        # The Python package (underscores)
│       ├── __init__.py           # Makes the folder importable
│       └── py.typed              # Optional type-checker marker
├── tests/                        # pytest finds these next week
├── pyproject.toml                # Package configuration
├── README.md                     # What people see on PyPI
└── .gitignore

Two names to keep apart:

  • PyPI name (in pyproject.toml): hyphenated, e.g. geohello-yourname
  • Python package name (folder under src/): underscored, e.g. geohello_yourname

pip install geohello-yourname then from geohello_yourname import hello.

pyproject.toml, the configuration hub

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "geohello-yourname"
version = "0.1.0"
description = "A simple geospatial greeting"
readme = "README.md"
requires-python = ">=3.10"
dependencies = []

Two sections. [build-system] says how to build. [project] says what the package is. We add dependencies and [project.optional-dependencies] later.

Live demo: four steps to a working package

We will build geohello together. Same commands you will run in 10 minutes.

# Step 1: scaffold IN your current folder (not in a subfolder)
mkdir geohello && cd geohello
uv init --lib --name geohello . # the 'period `.` ' is important!

# Step 2: edit src/geohello/__init__.py (one function)

# Step 3: build
uv build

# Step 4: try it (uv creates and manages .venv automatically)
uv run python -c "from geohello import hello; print(hello())"

uv handles the virtual environment for you — no manual setup needed.

  1. uv init — scaffolds the project (creates pyproject.toml, src/, etc.)
  2. uv add <package> — installs dependencies and updates pyproject.toml

uv automatically creates .venv/ on the first uv run or uv sync.

Step 1: scaffold with uv

mkdir geohello && cd geohello # only if you haven't
uv init --lib --name geohello .

The trailing . tells uv to scaffold in the current directory, not in a new subfolder. After this you have:

geohello/                       # <-- you are here
├── src/geohello/__init__.py
├── tests/
├── pyproject.toml
├── README.md
├── .python-version
└── .gitignore

uv filled in a working pyproject.toml, made the src/ layout, and will create .venv/ on the first uv run or uv sync. You did not have to remember any of it.

In your pyproject.toml (preventing from PyPi submission)

Open your pyproject.toml

[project]
classifiers = ["Private :: Do Not Upload"]

Step 2: write one function

Open src/geohello/__init__.py and replace the contents:

"""A simple geospatial greeting package."""

__version__ = "0.1.0"


def hello(place="Auckland"):
    """Return a greeting for a place."""
    return f"Hello from {place}!"

That is the entire package. One function, one docstring, one version.

Step 3: build the distributions

uv build

This creates a dist/ folder.

dist/
├── geohello-0.1.0-py3-none-any.whl    # the wheel
└── geohello-0.1.0.tar.gz              # the source archive

The wheel is what pip install actually downloads. The .tar.gz is a backup that contains your source code.

Step 4: try it

# Run the function in one line
uv run python -c "from geohello import hello; print(hello())"
# Hello from Auckland!

# Or open Python interactively
uv run python
>>> from geohello import hello
>>> hello("Mexico City")
'Hello from Mexico City!'

That is what “building a package” feels like. Four commands.

Common errors and fixes

  • ImportError: No module named geohello_yourname. Check the folder under src/ exists and uses underscores.
  • name 'geohello' is already taken (later, on TestPyPI). Add your initials to the name field.
  • Build fails on the first line of pyproject.toml. Usually a TOML syntax typo. Read the first error message carefully.
  • The function works in uv run python but not in plain python. You are not in the project’s virtual environment. Use uv run or source .venv/bin/activate.

Versioning conventions

Python packages use three numbers separated by dots:

MAJOR.MINOR.PATCH     e.g.  1.4.2
Bump What happened? Example
PATCH (1.4.2 → 1.4.3) Fixed a bug. Nothing new. Fixed a typo in an error message
MINOR (1.4.2 → 1.5.0) Added something new. Old code still works. Added a new load_weekly() function
MAJOR (1.4.2 → 2.0.0) Changed something existing. Old code might break. Renamed load_hourly() to fetch_hourly()

Your first release will be 0.1.0. When the major number is 0, it means “still experimenting — expect changes.”

github.com/dataandcrowd/akl-ped-counts

Part 7: Where does the data live?

Two patterns

Your A3 package needs data. Real Auckland data, not synthetic toys. Two patterns to choose from.

  • Bundle it. Ship a small file inside the package itself. Good for lookup tables, schemas, sensor lists, anything under about 10 MB.
  • Fetch it. Download from an API on first use, then cache locally. Good for large or frequently updated data such as Stats NZ census and LINZ boundaries.

akl-ped-counts uses the first pattern (8 MB of CSVs ship with the package).

Pattern 1: Bundling GIS data

Where to put the file

The convention is a data/ folder inside the package, not at the top level.

your-package/
├── pyproject.toml
├── README.md
└── src/
    └── your_package/
        ├── __init__.py
        ├── loader.py
        └── data/                # <-- here
            ├── sensors.csv
            └── zones.geojson

This is what akl-ped-counts does with hourly_counts.csv and locations.csv. Same idea, same place.

Finding the file at runtime: the problem

Your laptop knows where sensors.csv lives. Someone who runs pip install your-package does not.

# 🚫 Works on your laptop, breaks for everyone else
def load_sensors():
    return pd.read_csv("/Users/hyesop/projects/data/sensors.csv")

After pip install, the file lives somewhere inside their Python’s site-packages/... folder — a path you cannot predict.

Finding the file at runtime: the fix

Let Python find the package for you.

from importlib.resources import files
import pandas as pd

def load_sensors():
    path = files("your_package") / "data" / "sensors.csv"
    return pd.read_csv(path)

Read it as a sentence:

“Find the package your_package, go into its data/ folder, grab sensors.csv.”

The / here is the same idea as pathlib.Path — it joins folder names. No hard-coded /Users/... anywhere.

Same package works on your laptop, on a classmate’s laptop, on PyPI — Python figures out the real location.

Try it: loading Auckland pedestrian data

In src/geohello/__init__.py, add a loader next to hello():

from importlib.resources import files
import pandas as pd


def hello() -> str:
    return "Hello from geohello!"


def load_data():
    """Load the 2024 Auckland pedestrian counts."""
    path = files("geohello") / "data" / "akl_ped-2024.csv"
    return pd.read_csv(path)
uv run python -c "from geohello import load_data; print(load_data().shape)"

If you see the shape printed, the CSV is bundled and loading correctly.

Once you have more than one file

Repeating files("your_package") / "data" / ... gets noisy. Wrap it in a tiny helper.

from importlib.resources import files
import geopandas as gpd
import pandas as pd

DATA = files("your_package") / "data"   # computed once at import time

def load_sensors():
    return pd.read_csv(DATA / "sensors.csv")

def load_zones():
    return gpd.read_file(DATA / "zones.geojson")

That is the pattern akl-ped-counts uses. Two loaders, one DATA constant, no path strings to maintain.

Telling the build tool to include the file

Bundled data does not ship by default. You must tell hatchling to include it.

[tool.hatch.build.targets.sdist]
include = [
    "src/your_package/**",
    "README.md",
    "LICENSE",
]

[tool.hatch.build.targets.wheel]
packages = ["src/your_package"]

The ** glob picks up everything under src/your_package/, including the data/ folder. Without this, your CSVs are missing from the wheel and pip install looks fine but breaks at runtime.

Always sanity-check by unzipping the .whl after uv build. The data/ folder should be inside.

When to bundle vs not

  • Yes, bundle. Sensor metadata (a few KB). Schema files. Static reference tables. NZDep2023 score table (a few hundred KB).
  • Maybe bundle. A small CSV under 10 MB that does not change between releases.
  • No, do not bundle. Anything over 10 MB. SA2 boundary GeoPackages (often 100+ MB). LINZ canopy layers. Census shapefiles.

For the heavy stuff, fetch from an API.

Pattern 2: Fetching from the Stats NZ API

Why fetch instead of bundle

  • The data is too large to ship in a wheel.
  • The data updates every few months and you do not want to release a new version each time.
  • Stats NZ already hosts it. Stop re-hosting public data.

The cost is one API call on first use. The benefit is a tiny package and always-fresh data.

Stats NZ Datafinder, in plain language

Stats NZ runs a Koordinates-based portal at https://datafinder.stats.govt.nz/.

Two ways to download programmatically.

  • WFS (Web Feature Service). Standard OGC protocol. Returns GeoJSON. Best for vector layers and small bounding boxes.
  • Koordinates Export API. Asynchronous job, you request an export and poll until it finishes. Best for large or full-country pulls.

We will use WFS in the example because it is one HTTP call.

Step 1, get an API key

  1. Sign up at https://datafinder.stats.govt.nz/.
  2. Go to your profile, then “API keys”.
  3. Create a key and copy it. It looks like E5b....
  4. Save it somewhere safe.

You can do this during today’s hands-on. The free tier is plenty for any A3 package.

Step 2, never hard-code the key

Same pattern as the Week 4 lab: keep the key in a config.py next to your code, and add config.py to .gitignore.

# config.py  --  NEVER commit this file
API_KEY = "E5b..."
# loader.py
from config import API_KEY

Why a separate file? It is the simplest pattern that keeps your key out of the repo. Environment variables work too, but config.py is what you already used in Week 4 — stick with what is familiar.

Step 3, download all NZ regions

import geopandas as gpd
from config import API_KEY

LAYER_ID = 120945                          # Regional Council 2025 Clipped
WFS_URL = f"https://datafinder.stats.govt.nz/services;key={API_KEY}/wfs/"


def load_regions():
    """Download all 16 NZ regional council polygons."""
    params = (
        "service=WFS&version=2.0.0&request=GetFeature"
        f"&typeNames=layer-{LAYER_ID}"
        "&outputFormat=application/json"
        "&srsName=EPSG:2193"
    )
    return gpd.read_file(f"{WFS_URL}?{params}")

gpd.read_file accepts a URL directly — no requests boilerplate needed. The returned GeoDataFrame has 16 rows, one per region.

To try it interactively, open a Python shell with uv run python, then:

from geohello.loader import load_regions

gdf = load_regions()
gdf.shape
gdf.head()

Step 4a, inspect the schema first

Before writing a filter, look at what load_regions() actually returned. Never assume column names — Stats NZ schemas change every few years.

Option A — one-liner from the terminal:

uv run python -c "from geohello.loader import load_regions; gdf = load_regions(); print(gdf.columns.tolist())"

Option B — interactive Python shell:

uv run python
from geohello.loader import load_regions

gdf = load_regions()
gdf.columns.tolist()
gdf.iloc[0]
len(gdf)

Find the column that holds the region name (something like REGC2025_V1_00_NAME). Whatever it is on your machine, use that exact string in the next slide.

Step 4b, a function to filter Auckland

Now slice out just Auckland with an attribute filter.

NAME_COL = "REGC2025_V1_00_NAME"          # ← from Step 4a, edit if different


def filter_auckland(gdf):
    """Return rows whose region name contains 'Auckland'."""
    mask = gdf[NAME_COL].str.contains("Auckland", na=False)
    return gdf[mask].copy()

str.contains is forgiving — it matches whether Stats NZ stores the value as "Auckland" or "Auckland Region". If unsure, peek: print(gdf[NAME_COL].unique()).

Compose the two functions for a one-liner public API:

def load_auckland():
    """Convenience: fetch + filter in one call."""
    return filter_auckland(load_regions())

Why separate the filter from the download? Tests can call filter_auckland on a tiny fixture GeoDataFrame without ever hitting the network. Smaller, faster, offline-friendly tests.

Optional: server-side bbox (faster for Auckland-only)

If you only ever need Auckland, ask the server to clip during download.

AUCKLAND_BBOX = "1730000,5900000,1820000,5990000"   # EPSG:2193


def load_auckland_bbox():
    """Download only features inside the Auckland bounding box."""
    params = (
        "service=WFS&version=2.0.0&request=GetFeature"
        f"&typeNames=layer-{LAYER_ID}"
        "&outputFormat=application/json"
        "&srsName=EPSG:2193"
        f"&bbox={AUCKLAND_BBOX},EPSG:2193"
    )
    return gpd.read_file(f"{WFS_URL}?{params}")

Trade-off: bbox is faster and uses less bandwidth, but you lose the testable filter_auckland step. For A3 demos, prefer load_regions + filter_auckland — clearer to grade, easier to test.

Hands-on (10 min): try it once

With your partner, do the following.

  1. Sign up at https://datafinder.stats.govt.nz/ and copy your API key.
  2. In your package, create config.py with API_KEY = "..." and add config.py to .gitignore.
  3. Add load_regions() from the slide.
  4. First, inspect: print(gdf.columns.tolist()) — copy the actual name column into NAME_COL.
  5. Add filter_auckland() using your NAME_COL.
  6. Run uv run python -c "from geohello_yourname import load_regions, filter_auckland; print(filter_auckland(load_regions()).shape)".
  7. Confirm you get a GeoDataFrame with 1 row and a geometry column.

If it fails, check: (a) config.py sits next to the module that imports it, (b) NAME_COL matches what gdf.columns actually shows, (c) you are using https, not http.

Brief 1: pedestrian-exposure

Answer this question for a walker in Auckland: which route is healthier, and why?

  • What we expect to see: a CBD walking route scored on three environmental factors (noise, air pollution, green space), mapped together so the trade-offs are obvious.
  • The deliverable: a map of the route with each edge coloured by total exposure, a small bar chart breaking out the three components, and a one-line summary score.
  • The insight: which streets are quietest? Which are the most polluted? Where does greenery actually offset the others?
  • Key functions: add_exposure(route_gdf, layer, kind) and summarise_exposure(route_gdf).
  • Demo dataset: a route via OSMnx + OSM-based noise proxy + LAWA air quality + Sentinel-2 NDVI.
  • Difficulty: intermediate to ambitious (three small pipelines, kept consistent).

Brief 2: equitransport

Answer this question: does Auckland’s public transport access reach the most deprived neighbourhoods?

  • What we expect to see: a single number (Gini coefficient) plus a five-row quintile table that turns an equity argument into evidence.
  • The deliverable: a choropleth of Auckland SA2s coloured by NZDep quintile and a parallel map of PT access (stops within 400 m walk).
  • The insight: do the most deprived quintiles get fewer stops per capita? By how much? Where are the worst gaps?
  • Key functions: load_nzdep(sa2_gdf) and equity_summary(gdf).
  • Demo dataset: Stats NZ SA2 boundaries + AT GTFS PT-stop counts + NZDep2018 scores from the University of Otago.
  • Difficulty: intermediate (meshblock-to-SA2 aggregation is the tricky bit).

Brief 3: treecrown-nz

Answer this question: which Auckland neighbourhoods are tree-rich, and how shaded is a typical walk?

  • What we expect to see: a coverage number (% canopy) for each of three Auckland suburbs, and a route map coloured by overhead canopy fraction.
  • The deliverable: a side-by-side comparison plot for the three suburbs + a “shaded walk” map for one chosen route.
  • The insight: which suburbs have invested in canopy? Which routes stay shaded on a summer afternoon? Where is the canopy gap worst?
  • Key functions: canopy_coverage(area_gdf) and route_shade(route_gdf, buffer_m).
  • Demo dataset: Auckland Council LiDAR-derived canopy from the LINZ Data Service.
  • Difficulty: intermediate to ambitious (the LINZ API has a learning curve).

Brief 4: pavescore

Answer this question: can a phone photo tell us which Auckland pavements need fixing?

  • What we expect to see: every input image scored on a 0-to-100 condition scale with binary flags for cracking, surface degradation, and obstruction.
  • The deliverable: a map of scored photo locations along Dominion Road or Queen Street, coloured good / fair / poor, with a small table of the worst spots.
  • The insight: how does pavement quality vary along one street? Which segments are worst? Could a council prioritise repairs from this map?
  • Key functions: score_image(path_or_url) and to_geodataframe(results, coords).
  • Demo dataset: Mapillary street-level images.

Difficulty: ambitious, the most open-ended brief. A simple texture-based scorer that works beats an ambitious classifier that breaks.

Brief 5: escooter-akl

Answer this question: how do e-scooters (Flamingo) move around the CBD, and where are operators failing to enforce parking rules?

  • What we expect to see: a clean origin-destination flow map plus a count of trips that ended inside no-parking zones.
  • The deliverable: an OD chord diagram or arrow map between SA2 zones + a marker map of geofence violations with zone labels for reporting.
  • The insight: which CBD zones are origin-heavy vs destination-heavy? Where are violations concentrated? What story would you tell Auckland Council?
  • Key functions: od_flows(trips_gdf, zones_gdf) and geofence_violations(trips_gdf, no_park_gdf).
  • Demo dataset: a sample CSV of Auckland CBD e-scooter trips + Stats NZ SA2 zones + Auckland Council pedestrianised polygons.
  • Difficulty: intermediate (messy data; loud failure beats silent failure).
