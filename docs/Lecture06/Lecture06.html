Why Shiny?

Traditional web development (HTML + JavaScript):

document.getElementById("map").addEventListener("click",
  function(e) {
    // 50 lines of wiring code ...
  }
);

Shiny (Python):

@render.plot
def chart():
    return filtered().plot.bar(x="suburb", y="population")

Write Python, get a web app. Shiny handles HTML, CSS, JavaScript, and browser communication for you.

Shiny mental model

+------------------------------------------+
|  Browser (what the user sees)            |
|  +----------------+  +-----------------+ |
|  |    Inputs      |  |    Outputs      | |
|  |  [ Slider  ]   |  |   [ Map    ]    | |
|  |  [ Select  ]   |  |   [ Chart  ]    | |
|  |  [ Checkbox]   |  |   [ Table  ]    | |
|  +----------------+  +-----------------+ |
+------------------------------------------+
          |                    ^
          v                    |
+------------------------------------------+
|  Python server (your code)               |
|                                          |
|  @reactive.calc    -->  @render.plot     |
|  def filtered():        def chart():    |
|      ...                    ...         |
+------------------------------------------+

The user touches the left side. Your Python fills the right side. Shiny manages the arrows.

Three components of every Shiny app

from shiny import App, ui, render

# 1. UI: what the user sees
app_ui = ui.page_fluid(
    ui.h2("Hello Shiny"),
    ui.input_slider("n", "Pick a number",
                    0, 100, 50),
    ui.output_text("echo"),
)

# 2. Server: how outputs are computed
def server(input, output, session):
    @render.text
    def echo():
        return f"You picked {input.n()}"

# 3. App object: glues them together
app = App(app_ui, server)
Piece Job
UI Inputs + output placeholders
Server Python logic that fills the outputs
App Connects them so Shiny can run

That is a complete, runnable app.

The reactive loop

input.n()  (user moves slider)
    |
    v
echo()  recalculates automatically
    |
    v
Browser text updates

You did not write any code to detect the change.

Shiny noticed that echo reads input.n(), and that is enough.

This is reactive programming, the single idea that makes dashboards tractable.

The shape of a Shiny project

myapp/
+-- app.py              # the app itself
+-- pyproject.toml      # dependencies
+-- data/               # CSV, GeoJSON, GeoPackage
+-- www/                # static assets (images, CSS)
+-- README.md

Two conventions:

  • Anything in www/ is served at /your-file.png
  • Shiny looks for the variable called app. Rename it and Shiny will not find it.

Running the app

Three options (most common first):

  2. Terminal: shiny run --reload app.py
  3. Custom port: shiny run --port 8080 --reload app.py

--reload = restart on every save. Always use it while developing.

Stop with Ctrl+C.

Core vs Express syntax

Core (we use this)

app_ui = ui.page_fluid(
    ui.input_slider("n", "N", 0, 100, 50),
    ui.output_text("echo"),
)

def server(input, output, session):
    @render.text
    def echo():
        return f"{input.n()}"

app = App(app_ui, server)

Express (shorter, less control)

from shiny.express import input, render, ui

ui.input_slider("n", "N", 0, 100, 50)

@render.text
def echo():
    return f"{input.n()}"

Stick to Core. It scales better and matches what the R community expects.

Wiring inputs to outputs

def server(input, output, session):

    @render.text
    def message():
        count = input.count()
        if count > 500:
            return f"{count}: high traffic"
        return f"{count}: normal traffic"

Key rule: call inputs with parentheses.

input.hour     # the function object (wrong)
input.hour()   # the current value   (right)

Three kinds of reactive node

Kind Example Behaviour
Source input.x(), reactive.value() Changes from outside; no dependencies
Expression @reactive.calc Lazy, cached, returns a value
Observer @render.*, @reactive.effect Eager, runs for side effects

@reactive.calc: calculate once, reuse everywhere

@reactive.calc
def filtered():
    df = data[data["city"] == input.city()]
    return df[df["population"] >= input.min_pop()]
  • Define inside server
  • Call with parentheses: filtered()
  • Result is cached: call it ten times, runs once

Rule of thumb: if two outputs do the same filter, lift it into a @reactive.calc.

Without @reactive.calc (inefficient)

@render.plot
def plot1():
    data = load_and_process()   # slow!
    return make_plot(data)

@render.table
def table1():
    data = load_and_process()   # slow again!
    return data

With @reactive.calc (efficient)

@reactive.calc
def processed():
    return load_and_process()   # once!

@render.plot
def plot1():
    return make_plot(processed())   # reuses cache

@render.table
def table1():
    return processed()              # reuses cache

@reactive.effect: for things the user does not see

@reactive.effect
def log_year():
    print(f"[{input.year()}] user changed the year filter")

Same dependency tracking, but:

  • No return value
  • No caching
  • Runs for side effects (logging, writing a file)

If the user needs to see it, use @render.*, not @reactive.effect.

@reactive.event: “only when I say so”

@reactive.calc
@reactive.event(input.go)
def total():
    return input.a() + input.b()

Move the sliders: nothing happens.

Click “Recompute”: total updates.

Perfect for expensive computations where the user should control the timing.

Quick reference

Decorator Returns value? Cached? When does it run?
@reactive.calc Yes Yes When read, if stale
@reactive.effect No No Whenever deps change
@render.text / @render.plot Yes (to browser) No Whenever deps change
@reactive.event(input.x) (modifier) Only when input.x fires

Worked example: suburbs explorer

@reactive.calc
def filtered():
    df = data[data["city"] == input.city()]
    return df[df["population"] >= input.min_pop()]

@render.table
def tbl():
    return filtered()

@render.plot
def chart():
    df = filtered()
    fig, ax = plt.subplots(figsize=(7, 4))
    ax.barh(df["suburb"], df["median_income"],
            color="steelblue")
    ax.set_xlabel("Median income")
    return fig

@reactive.effect
@reactive.event(input.log)
def _():
    print(f"[LOG] {input.city()}, "
          f"n={len(filtered())} rows")

What happens:

  • Change city or slider: table + chart redraw
  • Click “Log”: message prints, nothing visible changes
  • filtered() runs once per change, shared by both outputs

Walk through the reactive graph

  • Two sources: input.city() and input.min_pop()
  • One expression: filtered(), cached and shared
  • Two render observers: tbl and chart, redraw when filtered changes
  • One effect observer: _, gated by @reactive.event(input.log), only the button triggers it

This pattern (shared calc, multiple render outputs, one gated effect) covers most dashboards you will ever build.

Two things to remember:

  • CRS: ipyleaflet needs WGS84 (EPSG:4326)
  • Layer refresh: return a new Map each time inside @render_widget

Debugging: four techniques that solve 90% of problems

1. Read the traceback (last line first)

AttributeError: 'Inputs' object has no attribute 'n'

You wrote input.n instead of input.n(), or the id does not match.

2. print() is your friend

@render.text
def echo():
    print(f"echo running with n={input.n()}")
    return f"You picked {input.n()}"

If it does not print, the function is not running.

Debugging (continued)

3. Silent failures = name mismatches

ui.output_text("echo")     # placeholder: "echo"

@render.text
def eccho():                # function: "eccho" (typo!)
    ...                     # blank output, no error

4. Test your functions outside Shiny first

# In a notebook or script
data = pd.read_csv("data/test.csv")
result = your_filter_function(data, "Auckland")
print(result.head())       # does it work?

If the function is correct, the Shiny wiring is the only suspect.

Deploying with shinylive: no server needed

Traditional web app:

User  -->  Server (running Python)  -->  Response
             (costs money, needs maintenance)

Shinylive (WebAssembly):

User  -->  Browser (runs Python via Pyodide)  -->  Response
             (free, no server, works offline)

Your app becomes static HTML files hosted on GitHub Pages for free.

Key takeaways

  1. A Shiny app has three pieces: UI, server, App
  2. Reactive programming: describe what depends on what, Shiny handles when
  3. Three reactive nodes: sources (inputs), expressions (@reactive.calc), observers (@render.*, @reactive.effect)
  4. @reactive.calc = cached shared value; @reactive.effect = side effects; @reactive.event = gated trigger
  5. ipyleaflet maps go inside Shiny via output_widget + @render_widget
  6. shinylive export . docs turns your app into a free static site
