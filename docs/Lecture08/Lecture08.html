Recap: the Shiny building blocks (Week 6)

Every Shiny app has the same three pieces:

from shiny import App, ui, render

app_ui = ui.page_sidebar(...)   # 1. UI: what the user sees
def server(input, output, session):
    @render.plot                # 2. Server: how outputs respond
    def chart(): ...

app = App(app_ui, server)       # 3. App: connects them

Reactivity is the key idea: when an input changes, any output that reads it re-runs automatically. You describe what depends on what, and Shiny handles when.

Recap: reactive nodes (Week 6)

Decorator Purpose Cached?
@reactive.calc Shared filtered data (call with ()) Yes
@reactive.effect Side effects (logging, writing files) No
@render.plot / @render.text Sends results to the browser No
@reactive.event(input.x) Only runs when input.x fires –

Rule of thumb: if two outputs need the same filtered data, lift it into a @reactive.calc.

The palmerpenguins dataset

The palmerpenguins package provides measurements for 344 penguins from three species, collected at Palmer Station, Antarctica. We have prepared a version with individual latitude/longitude coordinates for each penguin (jittered around island centroids).

These islands are in the Palmer Archipelago, off the Antarctic Peninsula (roughly 64 to 65 degrees S). Each penguin in our dataset has its own coordinate, so they spread out across each island rather than stacking on a single point.

How the coordinates were made

The original palmerpenguins package has no latitude/longitude. We created the dataset by:

  1. Looking up approximate centroids for each island (Torgersen: -64.77, -64.08; Biscoe: -65.43, -65.50; Dream: -64.73, -64.23)
  2. Jittering each penguin’s position by a small random offset (~500m) to simulate individual nest sites
# The idea behind the jittering:
base_lat, base_lon = island_coords[island]
lat = base_lat + np.random.uniform(-0.008, 0.008)
lon = base_lon + np.random.uniform(-0.012, 0.012)

For your Assignment 2, your data likely already has real coordinates, so you will not need this step.

Merging data: a pattern you will use

Even though our penguin CSV already has coordinates, the merge pattern is something you will use constantly in Assignment 2 when your attributes and geometries live in separate files:

# Your own version might look like:
stops = pd.read_csv("data/stops.csv")           # has stop_id, lat, lon
boardings = pd.read_csv("data/boardings.csv")   # has stop_id, count
combined = boardings.merge(stops, on="stop_id")  # now has geometry

The idea: one table has the what, another has the where. Merge them on a shared key.

Adding popups to markers

Popups let users click a marker and see information. Use the Popup widget or attach an HTML widget to a marker:

from ipyleaflet import Map, Marker, Popup
from ipywidgets import HTML

island_summary = penguins_geo.groupby("island").agg(
    n=("species", "size"),
    lat=("latitude", "mean"),
    lon=("longitude", "mean"),
).reset_index()

m = Map(center=(-64.8, -64.5), zoom=9, basemap=basemaps.Esri.WorldImagery)

for _, row in island_summary.iterrows():
    marker = Marker(location=(row["lat"], row["lon"]))
    popup_content = HTML(
        value=f"<b>{row['island']}</b><br>{int(row['n'])} penguins"
    )
    marker.popup = popup_content
    m.add(marker)

This is useful when each point has attributes you want to display on click, exactly like your Assignment 2 data.

From notebook to Shiny: the pattern

We have built the map step by step in a notebook. Now let us move it into a Shiny app. The pattern is always the same:

1. Get the code working in a notebook (static)
2. Identify the variable the user should control
3. Replace that variable with an input widget
4. Wrap the output in a render function
5. Done: Shiny handles the rest

For our penguin map, the variable the user should control is species.

Step 1: static code (already done)

We already have this working:

species = "Adelie"   # <-- the variable the user will control
sel = penguins_geo[penguins_geo["species"] == species]

m = Map(center=(-64.8, -64.5), zoom=9, basemap=basemaps.Esri.WorldImagery)
# ... add markers for sel ...

Step 2: add an input widget

app_ui = ui.page_sidebar(
    ui.sidebar(
        ui.input_radio_buttons(
            "species", "Species",
            choices=["Adelie", "Gentoo", "Chinstrap"],
        ),
    ),
    output_widget("map"),
    title="Palmer Penguins Explorer",
)

The id is "species", so in the server we read it with input.species().

Step 3: wrap in a render function

def server(input, output, session):
    @render_widget
    def map():
        species = input.species()                          # <-- reactive
        sel = penguins_geo[penguins_geo["species"] == species]

        m = Map(center=(-64.8, -64.5), zoom=9,
                basemap=basemaps.Esri.WorldImagery)
        for _, row in sel.iterrows():
            m.add(CircleMarker(
                location=(row["latitude"], row["longitude"]),
                radius=8, color="#447099",
                fill_color="#447099", fill_opacity=0.7,
            ))
        return m

Every time the user picks a different species, map() re-runs and the browser updates.

The complete app

from shiny import App, ui, render, reactive
from shinywidgets import output_widget, render_widget
from ipyleaflet import Map, CircleMarker, basemaps
import pandas as pd

penguins_geo = pd.read_csv("data/penguins_geo.csv")

app_ui = ui.page_sidebar(
    ui.sidebar(
        ui.input_radio_buttons(
            "species", "Species",
            choices=["Adelie", "Gentoo", "Chinstrap"],
        ),
    ),
    ui.output_text("summary"),
    output_widget("map"),
    title="Palmer Penguins Explorer",
)

def server(input, output, session):
    @reactive.calc
    def filtered():
        return penguins_geo[penguins_geo["species"] == input.species()]

    @render_widget
    def map():
        sel = filtered()
        m = Map(center=(-64.8, -64.5), zoom=9,
                basemap=basemaps.Esri.WorldImagery)
        for _, row in sel.drop_duplicates("island").iterrows():
            m.add(CircleMarker(
                location=(row["latitude"], row["longitude"]),
                radius=12, color="#447099",
                fill_color="#447099", fill_opacity=0.7,
            ))
        return m

    @render.text
    def summary():
        sel = filtered()
        n = len(sel.dropna(subset=["bill_length_mm"]))
        avg_mass = sel["body_mass_g"].mean()
        return (f"{input.species()}: {n} penguins, "
                f"avg body mass {avg_mass:,.0f} g")

app = App(app_ui, server)

Notice the @reactive.calc

@reactive.calc
def filtered():
    return penguins_geo[penguins_geo["species"] == input.species()]

Both map() and summary() call filtered(). The filtering runs once per species change, not twice.

This is the same pattern from Week 6, and the same pattern you should use in Assignment 2.

Adding an action button with @reactive.event

Sometimes you do not want every change to trigger an update immediately, especially if the computation is expensive. An action button lets the user adjust several inputs first, then click once:

# In app_ui, inside ui.sidebar():
ui.input_action_button("apply", "Apply filters"),

# In server:
@reactive.calc
@reactive.event(input.apply, ignore_none=False)
def filtered():
    df = penguins_geo[penguins_geo["species"] == input.species()]
    return df[df["island"].isin(input.islands())]

Now filtered() only recalculates when the user clicks “Apply filters”, not on every radio button or checkbox change.

When to use @reactive.event

Use it when:

  • The user adjusts multiple inputs before wanting results (avoids flickering)
  • The computation is slow (large dataset, network call)
  • You want the user to confirm before the dashboard updates

Do not use it when:

  • You only have one input (the button adds unnecessary friction)
  • The computation is fast and instant feedback is helpful

For Assignment 2, if your filtering is quick, you probably do not need @reactive.event. If loading or processing takes more than a second, consider it.

Adding a chart alongside the map

from shiny import render

# In app_ui, add after output_widget("map"):
ui.card(
    ui.card_header("Bill length distribution"),
    ui.output_plot("hist"),
),

# In server:
@render.plot
def hist():
    import matplotlib.pyplot as plt
    sel = filtered()
    fig, ax = plt.subplots(figsize=(7, 3))
    ax.hist(sel["bill_length_mm"].dropna(), bins=15,
            color="#447099", edgecolor="white")
    ax.set_xlabel("Bill length (mm)")
    ax.set_ylabel("Count")
    return fig

Map and chart share filtered(). One @reactive.calc, two outputs.

Using GeoData with a GeoDataFrame

For Assignment 2, most of you will work with polygons or lines (SA2 boundaries, bus routes, crash zones), not point markers. Use GeoData instead of CircleMarker:

from ipyleaflet import Map, GeoData
import geopandas as gpd

gdf = gpd.read_file("data/sa2.geojson").to_crs(4326)  # always reproject!

m = Map(center=(-36.85, 174.77), zoom=11)
m.add_layer(GeoData(
    geo_dataframe=gdf,
    style={"color": "#d73027", "weight": 2, "fillOpacity": 0.4},
    hover_style={"fillColor": "yellow", "fillOpacity": 0.6},
    name="SA2 boundaries",
))

hover_style gives visual feedback when the user moves their mouse over a polygon, no extra code needed.

GeoData inside Shiny

The pattern is identical to what we did with CircleMarker:

@reactive.calc
def filtered_gdf():
    gdf = all_data[all_data["category"] == input.category()]
    return gdf.to_crs(4326)

@render_widget
def map():
    m = Map(center=(-36.85, 174.77), zoom=11)
    m.add_layer(GeoData(
        geo_dataframe=filtered_gdf(),
        style={"color": "#1565c0", "weight": 2, "fillOpacity": 0.4},
        hover_style={"fillColor": "yellow", "fillOpacity": 0.6},
    ))
    return m

Remember: return a new Map() each time. Do not try to mutate a map created at module level.

Two reminders for ipyleaflet

  1. Always .to_crs(4326) when loading a GeoDataFrame. ipyleaflet expects WGS84. If your data is in NZTM (EPSG:2193), the map will appear empty with no error.

  2. Return a new Map() inside @render_widget every time. Do not create the map at module level and try to mutate it.

# Wrong:
m = Map(...)           # created once at module level
@render_widget
def map():
    m.add_layer(...)   # tries to mutate, will not update
    return m

# Right:
@render_widget
def map():
    m = Map(...)       # new map each time
    m.add_layer(...)
    return m
