A simple module to test

Before we write tests, we need something to test. Add this function to your __init__.py, below hello().

import pandas as pd

def filter_by_month(df, month: int):
    """Return rows for a given month (1-12)."""
    dates = pd.to_datetime(df["Date"], errors="coerce")
    return df[dates.dt.month == month]

Create demo_filter.py in your project root:

# demo_filter.py
from geohello import load_data, filter_by_month

df = load_data()
print("All rows:", len(df))
print("March only:", len(filter_by_month(df, month=3)))
uv run python demo_filter.py
# All rows: 8795
# March only: 744

Now you have two functions worth testing: hello() and filter_by_month().

Part 1: pytest in three commands

What is an assertion in testing?

An assertion confirms that a programme functions as intended by examining whether specific conditions are met during execution.

Each test should have only one reason to fail or succeed.

assert 2 + 2 == 4        # passes — condition is True
assert 2 + 2 == 5        # fails  — condition is False

If assert passes, nothing happens. If it fails, pytest reports exactly which line failed and what the values were.

All you need to write a test

Three things.

  1. A file under tests/ whose name starts with test_.
  2. A function inside it whose name starts with test_.
  3. An assert line that says what should be true.
# tests/test_basics.py
def test_two_plus_two():
    assert 2 + 2 == 4

That is a complete, runnable test.

Three commands

# Install pytest as a dev dependency
uv add --dev pytest

# Run every test in the project
uv run pytest

# Verbose output so you see each test name
uv run pytest -v

Without -v — dots only:

tests/test_basics.py .                                                   [100%]

With -v — each test name and result:

tests/test_basics.py::test_two_plus_two PASSED                           [100%]

-v stands for verbose. Use it so you can see exactly which test passed or failed.

A test for geohello

# tests/test_geohello.py
from geohello import hello


def test_hello_default():
    result = hello()
    assert "Auckland" in result


def test_hello_custom_place():
    assert hello("Tokyo") == "Hello from Tokyo!"


def test_hello_returns_string():
    assert isinstance(hello(), str)

What happens when a test fails?

def test_hello_wrong_city():
    result = hello()
    assert "Wellington" in result   # wrong on purpose
FAILED test_fail_demo.py::test_hello_wrong_city

    def test_hello_wrong_city():
        result = hello()
>       assert "Wellington" in result
E       AssertionError: assert 'Wellington' in 'Hello from Auckland!'

pytest shows the exact line, the expected value, and the actual value. No guessing.

A note on test design

  • One assertion per test, by default. Failures should pinpoint exactly what is wrong.
  • Names describe the behaviour, not the function. test_hello_default beats test_function_1.
  • Keep tests tiny. A test that takes 2 minutes will not get run.

Part 2: Three patterns you will actually use

When do I use each pattern?

Pattern When to use Example
CRS guard Your function loads or returns spatial data Check the output is in EPSG:2193
Fixture Multiple tests need the same input data Create a small DataFrame once, reuse in 5 tests
Parametrise Same logic, different inputs Test hello() with Auckland, Tokyo, Wellington

Pattern 1: a CRS guard

For most A3 briefs, your spatial maths needs to be in EPSG:2193 (NZTM, metres). One test catches the most common mistake.

def test_output_has_correct_crs():
    result = function_one(...)
    assert result.crs.to_string() == "EPSG:2193"

If function_one ever forgets to reproject, the test fails. You see it in seconds, not weeks.

Pattern 2: a fixture

A fixture is reusable test data, set up once and shared across tests.

import pytest
import pandas as pd
from geohello import filter_by_month


@pytest.fixture
def sample_df():
    return pd.DataFrame({
        "Date": ["2024-01-15", "2024-01-20", "2024-03-10"],
        "Count": [100, 200, 150],
    })


def test_filter_returns_only_january(sample_df):
    result = filter_by_month(sample_df, month=1)
    assert len(result) == 2


def test_filter_empty_month(sample_df):
    result = filter_by_month(sample_df, month=7)
    assert len(result) == 0

Pattern 3: parametrize

Run the same test with several inputs.

@pytest.mark.parametrize("place,expected", [
    ("Auckland", "Hello from Auckland!"),
    ("Tokyo", "Hello from Tokyo!"),
    ("Wellington", "Hello from Wellington!"),
])
def test_hello_for_many_places(place, expected):
    assert hello(place) == expected

One test definition, three test runs. Very useful for “this should work for any of these CRS / years / sensor IDs”.

Part 3: TestPyPI rehearsal

Why TestPyPI first

  • Practice the publication workflow safely.
  • Catch missing files and broken metadata early.
  • Test the install in a clean environment.
  • Packages are deleted after 6 months, so mistakes do not haunt you.

Do this this week, before the real PyPI publish in Week 12.

The four steps

# 1. Bump the version (TestPyPI rejects duplicates)
# Edit pyproject.toml: version = "0.1.1"

# 2. Build
uv build

# 3. Publish (you will be prompted for the token)
uv publish --publish-url https://test.pypi.org/legacy/

# 4. Install in a clean venv to confirm
uv venv /tmp/test-install
source /tmp/test-install/bin/activate

uv pip install \
    --index-url https://test.pypi.org/simple/ \
    --extra-index-url https://pypi.org/simple/ \
    your-package-name

The --extra-index-url is needed because pandas, geopandas, and friends only live on real PyPI.

Step 1: Create a TestPyPI account

  1. Go to https://test.pypi.org/account/register/.
  2. Fill in username, email, password.
  3. Verify your email (check spam).

Step 2: Generate an API token

  1. Log in to https://test.pypi.org/.
  2. Click your username (top right) → Account settings.
  3. Scroll down to API tokens → click “Add API token”.
  4. Token name: uv-publish (or anything you like).
  5. Scope: “Entire account” (first time; you can scope to a project later).
  6. Click “Create token”.
  7. Copy the token now — it starts with pypi- and is shown only once.

Step 3: Publish with the token

# Build first
uv build

# Publish — paste your token when prompted for password
uv publish --publish-url https://test.pypi.org/legacy/

When prompted:

  • Username: __token__
  • Password: paste the pypi-... token

Or skip the prompt by setting an environment variable:

export UV_PUBLISH_TOKEN=pypi-YOUR_TOKEN_HERE
uv publish --publish-url https://test.pypi.org/legacy/

Errors you will hit and how to fix them

  • The name is already taken. Rename in pyproject.toml, add initials.
  • File already exists. Bump version (0.1.0 becomes 0.1.1).
  • Invalid or expired token. Generate a new token.
  • Could not find a version that satisfies the requirement pandas. Add --extra-index-url https://pypi.org/simple/ to the install command.

These are not rare bugs. Every cohort sees all four. Knowing the fix in advance saves you 30 minutes each.

Part 4: Poster design

The poster showcase

  • When, Monday 25 May.
  • Where, B301-G10.
  • What, A2 portrait poster, printed.
  • Format, two-minute pitch from each pair, then circulating Q&A.

Today you draft the layout on paper. Polish happens next week.

A2 size, in plain language

A2 is 240 mm by 594 mm. Read from two metres away.

  • Title text, 50 pt or larger.
  • Body text, 20 pt or larger.
  • One dominant figure, not five small ones.
  • Plenty of white space.
  • Most weak posters have too much text.

The seven-box rule

Take an A4 sheet. Divide into seven labelled regions.

Box Content
1. Title Package name, both authors
2. The problem Two sentences, no jargon
3. The solution One sentence stating what the package does
4. Code example Six lines showing the simplest use
5. Figure Map, chart, or table from the demo notebook
6. Design choices Three bullets (CRS choice, data choice, scope)
7. Links TestPyPI URL, GitHub URL, contact

If you cannot fill a box, you do not yet know what goes there. That is the point.

A common failure mode

Posters that try to explain everything look like dense walls of text. Nobody reads them.

A working poster answers three questions in 30 seconds.

  1. What problem does this solve?
  2. What does the package look like in practice?
  3. Where can I find it?

Everything else is for the conversation, not the poster.

Part 5: Fit interview, Tomorrow in the lab

What the fit interview is

In today’s lab, each pair member sits a short individual interview, 5 minutes per person.

  • “Walk me through one function you (personally) wrote. What does it return?”
  • “Why did you choose this CRS, this default, or this optional dependency?”
  • “What would happen if I passed an empty GeoDataFrame to this function?”

Bring your laptop with the package open. The interview is not a memory test and it is not a final exam. It is a quick check that both partners are contributing and that you understand the code your name is on.

How it runs in the lab

Pairs cluster into groups of four. Each pair interviews the other pair, one person at a time, while a demonstrator floats between groups to listen in.

  • One person sits, one person from the other pair asks the three questions.
  • Three to five minutes per person, four people total per group, around 20 minutes.
  • The demonstrator sits in on at least one interview per group.
  • Swap so all four people get a turn.

Why now, in Week 10? So uneven contribution can be caught and corrected before Week 11 polish and Week 12 PyPI release.

What we are listening for

  • Can you describe what the function returns and why?
  • Can you defend one design choice instead of just listing what you did?
  • When you do not know, can you say “I do not know, but I would check…”?

Hesitating, looking at your screen, or saying “let me check the docstring” are all fine. Saying “my partner wrote that bit and I am not sure how it works” is what we are listening to catch early.
